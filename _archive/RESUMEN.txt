---------------------------------------------------------------------------
DEF #1: EL CATÁLOGO SE USA LITERAL, NO COMO INSPIRACIÓN (crítica)
---------------------------------------------------------------------------
El flujo search_catalog.py → .section.json → compose_page.php inyecta el
archivo completo del catálogo en el page-def con TODOS sus estilos originales
embebidos (imágenes de demo, colores mapeados por RGB forzado, spacing).

Evidencia en acerca-de.json:76:
  "url": "https://diviplus.io/wp-content/uploads/2019/11/day_care_img_16.png"
  → Imagen de guardería en sitio de biblioteca académica

Evidencia en acerca-de.json:13:
  "color": "{{design:color:success}}"  → Verde éxito sobre diseño sepia

El orquestador (orchestrate_page.php:296-300) PREFIERE el catálogo sobre las
bases locales si el score semántico > 0.60. Pero las bases locales tienen
diseño premium con variantes (editorial-grid, liquid-glass, monochrome-brutalist)
mientras que el catálogo trae estilos genéricos de demos Divi 4.

Ruta crítica: orchestrate_page.php:268-310, compile_catalog.py:180-255

---------------------------------------------------------------------------
DEF #2: COLOR MAPPING RGB — PERCEPTUALMENTE INCORRECTO
---------------------------------------------------------------------------
compile_catalog.py:118-133 usa distancia euclidiana en espacio RGB.
RGB no es perceptualmente uniforme — el ojo humano no percibe diferencias
lineales en RGB. CIEDE2000 o al menos Lab sería correcto.

Un color #4A7C59 (verde bosque) puede quedar más cerca de "success" que de
"sepia-500" en RGB, aunque visualmente no tenga relación. Esto produce:
  - Fondos verdes en sitios sepia
  - Colores de acento incorrectos
  - Paletas rotas sistemáticamente

Ruta crítica: compile_catalog.py:107-158

---------------------------------------------------------------------------
DEF #3: DOS PARADIGMAS EN GUERRA DENTRO DEL MISMO PAGE-DEF
---------------------------------------------------------------------------
Paradigma Premium (home.json):
  "presets": ["section:hero-dark", "text:display-xl", "module:glass-card"]
  Usa tokens {{design:color:*}}, decoration semántica, responsive spacing,
  animaciones encadenadas, variantes de diseño.

Paradigma Legacy (acerca-de.json):
  "type": "regular", "module": "divi/section"
  "type": "divi/placeholder", "module": "divi/placeholder"
  Estilos embebidos del catálogo, spacing en %, imágenes de demo,
  color mapping forzado, placeholders Divi 4 no migrados.

El orquestador mezcla ambos porque elige entre catálogo (legacy) y base
(premium) según threshold semántico, SIN considerar calidad del diseño.

---------------------------------------------------------------------------
DEF #4: SOLO 3 VARIANTES DE DECORACIÓN PARA 14+ TIPOS DE SECCIÓN
---------------------------------------------------------------------------
Variantes existentes:
  - hero-split@editorial-grid     (asymmetric 3_5,2_5 + stagger + reveal)
  - hero-centered@liquid-glass    (glassmorphism + fluid)
  - features-3col@monochrome-brutalist (0 radius + high contrast)

Tipos de sección SIN variantes premium:
  stats-4col, testimonials-3col, cta-centered, content-split,
  content-split-icon-list, logos-4col, contact, about, gallery, faq

→ Son .section.json standalone con decoration genérica. No compiten con
  el nivel "premium" que el pipeline debería garantizar.

Ruta: workspace/sections/ (14 archivos, solo 4 con subdirectorio)

---------------------------------------------------------------------------
DEF #5: compose_page.php NO VALIDA NADA — SLOTS HUÉRFANOS
---------------------------------------------------------------------------
compose_page.php:124-143 → fill_slots() reemplaza slots no resueltos con "".
Esto produce en page-defs:
  - divi/button con button_text: "" → botón invisible (acerca-de.json:152)
  - divi/image con src: "" → imagen rota (acerca-de.json:293-294)
  - divi/heading sin contenido → título vacío (modern-test.json:137-138)
  - Slots {{slot:role}} y {{slot:avatar}} sin resolver en testimonials

No hay validación post-composición, ni asignación de decoration por defecto.

---------------------------------------------------------------------------
DEF #6: design-patterns.json NO SE APLICA (dead weight)
---------------------------------------------------------------------------
extract_patterns.py analiza 892 templates y produce design-patterns.json
(~10KB). Pero orchestrate_page.php:77-85 solo lo LOGUEA, nunca modifica
la composición real basada en estos patrones. Es código muerto.

Ruta: workspace/design-patterns.json → orchestrate_page.php:68-86

---------------------------------------------------------------------------
DEF #7: NO HAY CAPA DE "DISEÑO PREMIUM POST-COMPOSICIÓN"
---------------------------------------------------------------------------
El pipeline termina en compose_page.php → page-defs/ → build_page.php.
NO hay un paso intermedio que:
  - Elimine decoration heredada del catálogo
  - Inyecte presets de la marca en módulos que no los tienen
  - Reemplace imágenes de demo con assets abstractos de alta calidad
  - Valide coherencia visual contra el tone del brief

---------------------------------------------------------------------------
DEF #8: generate_brief.py NO USA EL DESIGN SYSTEM REAL
---------------------------------------------------------------------------
generate_brief.py:114-159 tiene SYSTEM_PROMPT hardcodeado para Bibliotheca
("sepia/parchment colors", "Cormorant Garamond"). Para otra marca, el LLM
genera briefs inconsistentes. Además la IA no sabe qué section_type tienen
template asociado — puede generar "pricing" o "timeline" que no existen.

Ruta: workspace/automation/generate_brief.py

---------------------------------------------------------------------------
DEF #9: search_catalog.py ES UN CAÑÓN PARA MOSQUITOS
---------------------------------------------------------------------------
Dependencia de SentenceTransformers + torch CPU (~500MB) para hacer
búsqueda coseno contra 892 embeddings de 384 dimensiones. Se resuelve
con numpy puro + embeddings precomputados sin torch.

Además el modelo all-MiniLM-L6-v2 es genérico, no fine-tuned para
terminología de diseño web ni dominios específicos.

Ruta: workspace/automation/search_catalog.py + requirements.txt

---------------------------------------------------------------------------
DEF #10: EL PAGE-DEF NO GARANTIZA DISEÑO PREMIUM (la clave del problema)
---------------------------------------------------------------------------
El contrato fundamental es: "page-def perfecto → despliegue perfecto".
Pero hoy un page-def puede contener:
  - divi/placeholder → cajas vacías en frontend
  - Colores mapeados incorrectamente → paleta rota
  - Imágenes de catálogo ajeno → contexto visual equivocado
  - Slots sin resolver → módulos huérfanos
  - Presets inexistentes (text:eyebrow-light) → silent failure
  - Spacing hardcodeado del catálogo → inconsistencias responsive

NO EXISTE UN ESTÁNDAR MÍNIMO DE CALIDAD PARA PAGE-DEF.

2.4 MAPA DE CONCEPTOS CLAVE
==================================

TÉRMINO              | DESCRIPCIÓN
---------------------|----------------------------------------------------
page-def             | ★ JSON semántico de entrada (tokens {{design:*}}, presets)
                     |   Es el CONTRATO: si es perfecto, el deploy es perfecto
                     |   Ruta: site/<marca>/page-defs/<slug>.json
.page.json           | Composición intermedia (brief → templates + slots)
                     |   Ruta: site/<marca>/compositions/<slug>.page.json
brief                | YAML de entrada con tone + secciones + slots
                     |   Ruta: site/<marca>/briefs/<slug>.yml
presets              | 64 templates de decoration reutilizables
                     |   Categorías: section, text, module, divider,
                     |   animation, scroll, transform
                     |   Ruta: brand/_design_presets.json (1868L)
tokens               | {{design:color:name}}, {{design:font:name}},
                     |   {{design:radius:name}}, {{design:space:name}}
                     |   Resuelto en build_page.php → var(--gcid-*) o literal
variants             | Decoraciones que sobreescriben _base.section.json
                     |   Sintaxis: template@variant (ej. hero-split@editorial-grid)
                     |   Solo 3 existentes: editorial-grid, liquid-glass,
                     |   monochrome-brutalist
gcid                 | Global Colors ID (var(--gcid-*)) en Divi 5
                     |   Sincronizar con: wp agentic global_colors sync
tone                 | Dirección visual: editorial, modern, premium, minimal,
                     |   dramatic, playful
                     |   Determina qué variante de decoración se aplica
DAW_SITE             | Variable de entorno para seleccionar marca activa
                     |   Default: bibliotheca

==================================
2.9 ERRORES CONOCIDOS Y PITFALLS
==================================

1. number-counter crash: Si title es objeto en vez de string, strpos()
   en layout-engine.php:436 crashea. Fix: if(is_array($title_text)) fallback.
   Documentado en workspace/sections/STATS.md

2. Presets inexistentes: build_page.php no valida que un preset exista.
   get_preset() devuelve [] silentemente. Ej: text:eyebrow-light no existe
   pero se referencias sin error en modern-test.json:293

3. Columnas sin type en catálogo: compile_catalog.py no siempre asigna
   column_structure correctamente en rows anidados. layout-engine.php
   infiere desde advanced.type desktop.value.

4. Gradient stops con % duplicado: build_page.php normaliza quitando %
   pero compile_catalog.py los deja con %. Layout_Engine también normaliza.
   Error si ambos normalizan → pierde información.

5. YAML parser manual sin librería: orchestrate_page.php:96-244 tiene
   parser YAML propio. No soporta YAML complejo (anidamiento profundo,
   arrays inline, referencias). Si el brief YAML se desvía del formato
   esperado, el parser falla silenciosamente.

6. search_catalog.py sin GPU: Force CPU con CUDA_VISIBLE_DEVICES="".
   En hardware sin CUDA es correcto, pero en máquinas con GPU desperdicia
   capacidad de cómputo.

7. Deep merge array indexed vs asociativo: compose_page.php:47 usa
   array_keys($value) !== range(0, count($value)-1) para detectar arrays
   indexados. Falla si los keys numéricos no son 0-indexed consecutivos.

==================================
2.10 REGLAS DE OPERACIÓN (desde AGENTS.md)
==================================
1. No editar divitheme.json a mano — regenerar con build_design_system.py
2. No usar divi/code como comodín — consultar blocks-dictionary.md
3. No usar et_pb_* (shortcodes Divi 4) — solo namespace divi/*
4. Colores siempre como {{design:color:*}}, nunca hex hardcodeados
5. build_page.php con resolved=true expande presets inline
6. Sin CSS inyectado en functions.php ni overrides en style.css
7. Posiciones de gradient sin % (Layout Engine normaliza)
8. Frontera site/: datos de proyecto separados del framework DAW
9. Pipeline 100% PHP para crear páginas (sin Python)
10. Multi-marca: cambiar $env:DAW_SITE para alternar proyectos
11. No usar mysql_remote.bat hasta revalidar conexión remota
