  El pipeline PHP es sólido. El problema siempre estuvo en el INPUT:
  el agente ejecutaba las 4 fases como checklist mental sin producir
  artefactos entre ellas. La Fase 2 (Design Lead) se saltaba o se hacía

    + Anti-patrón añadido: divi/text para H1 (debe ser divi/heading)
    + Ejemplo completo de Design Brief con las 6 leyes confirmadas

  Lo que NO cambió:
    - build_page.php (sólido, no tocar)
    - El pipeline PHP completo (funciona)
    - Los presets, tokens, design system (sin cambios)
    - La estructura de directorios

FASE 2 — DESIGN LEAD (⛔ BLOQUEANTE — las 6 leyes deben estar en verde)
─────────────────────────────────────────────────────────────────────────────
Referencia: DAW_bundle/daw-skill/references/design-lead.md (leer COMPLETO)

Las 6 Leyes de Calidad Autónoma (cada una es un bloqueante):

  LEY 1 — ALTERNANCIA DE FONDOS
    Nunca 2 secciones consecutivas del mismo color de fondo.
    Documentar la secuencia completa ANTES de escribir JSON:
      sec-1: section:hero-dark
      sec-2: section:light
      sec-3: section:white
      sec-4: section:dark
      ...
    Si el plan tiene secciones iguales consecutivas → DETENER y corregir.

  LEY 2 — TITULAR DOMINANTE
    H1 del hero SIEMPRE con divi/heading + preset text:display-xl o text:hero-title.
    72px mínimo en desktop. Sin esto no hay hero, hay una sección genérica.
    Si el plan no lo incluye → AGREGAR uno antes de continuar.

  LEY 3 — ESPACIO NEGATIVO MÍNIMO
    Secciones: padding top/bottom mínimo {{design:space:2xl}} (96px)
    Cards: padding interno mínimo {{design:space:lg}} (40px)
    Declararlo en decoration.spacing de cada sección y card en el JSON.

  LEY 4 — MICRO-INTERACCIÓN EN ELEMENTOS CLICKEABLES
    Documentar hover de CADA elemento interactivo antes del handoff:
      Botones primarios  → module:btn-primary + transform:hover-lift
      Feature cards      → module:feature-card + transform:hover-lift
      Glass cards        → module:glass-card + transform:hover-lift
      Imágenes enlazadas → transform:hover-scale
    Si el plan tiene tarjetas sin hover → AGREGAR preset de transform.

  LEY 5 — ANCLAJE VISUAL POR SECCIÓN
    Cada sección tiene UN elemento dominante identificado:
      Hero         → titular display-xl
      Stats        → número en stat-num (48px+)
      Features     → ícono del blurb
      Testimonial  → cita en quote-serif
      CTA          → botón grande aislado con padding masivo
    Si una sección no tiene ancla → DEFINIR una antes de continuar.

  LEY 6 — ESCALA RESPONSIVA DECLARADA
    Tipografía hero con los 3 breakpoints:
      H1: desktop 72px / tablet 44px / mobile 30px
      H2: desktop 40px / tablet 30px / mobile 24px
    El Diseñador implementa en headingFont.desktop/tablet/phone.value.size.

Artefacto obligatorio — Design Brief:
  === DESIGN BRIEF: <nombre de página> ===
  Estilo visual: <editorial / modern / premium / minimal>
  ALTERNANCIA DE FONDOS (Ley 1 ✓): sec-1: X / sec-2: Y / ...
  TITULAR HERO (Ley 2 ✓): divi/heading, preset: text:hero-title
  ESPACIO NEGATIVO (Ley 3 ✓): secciones 96px / cards 40px
  HOVER (Ley 4 ✓): botones → hover-lift / cards → hover-lift
  ANCLAS (Ley 5 ✓): hero → titular / stats → números / ...
  ESCALA RESPONSIVA (Ley 6 ✓): H1 72/44/30px / H2 40/30/24px
  DECISIONES: <decisión>: <fundamento>
  RESTRICCIONES: no hex / no placeholder / no catálogo como estructura

⛔ STOP: sin Design Brief escrito con las 6 leyes en verde, no comenzar Fase 3.

  .\php.bat DAW_bundle/divi-agentic-core/bin/build_page.php ^
    --def=DAW_bundle/site/bibliotheca/page-defs/<slug>.json ^
    --out=DAW_bundle/site/bibliotheca/pages/<slug>.json
  → Verificar: tokens resueltos, presets expandidos, sin {{design:*}} restantes

Lo que hace build_page.php en orden:
  1. Carga divitheme.json (design system)
  2. Carga schemas de módulos (workspace/data/modules/ — 102 archivos)
  3. Resuelve {{design:color:name}} → var(--gcid-name)
  4. Resuelve {{design:font|radius|space:name}} → literales
  5. Resuelve {{SITE_URL}} → URL del sitio
  6. Expande presets[] inline via deep_merge()
  7. Normaliza gradient stops (quita %)
  8. Valida jerarquía: sections → rows → columns → modules
  9. Con --deploy: wp agentic deploy_page → Layout Engine → bloques Divi 5
  10. Con --verify: verify_page.php comprueba gcids, estructura, tokens

Artefacto obligatorio:
  WP Post ID: <id>
  URL confirmada: <http://...>
  Verificación: bloques Divi 5 ✓ / gcids resueltos ✓ / sin tokens sin resolver ✓

⛔ STOP: si el deploy falla o la verificación reporta errores, diagnosticar
         y re-ejecutar antes de declarar la tarea completa.

SECCIÓN 4 — VERIFICACIÓN POST-DEPLOY
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

verify_page.php comprueba automáticamente (con --verify):
  ✓ La página existe en WordPress (wp_posts)
  ✓ El contenido contiene bloques Divi 5 (<!-- wp:divi/*)
  ✓ Variables var(--gcid-*) presentes
  ✓ Sin tokens {{design:*}} sin resolver
  ✓ Estructura coincide con schema (si se pasa --schema)

Verificación manual:
  1. WP Admin → Páginas → buscar slug → tiene contenido (no vacía)
  2. Abrir URL en navegador → renderiza con Divi y diseño correcto
  3. Si colores incorrectos: ejecutar sync de global colors (Paso 1.5)

Errores comunes:
  - Página en blanco: Layout Engine falló → revisar plugin activo, revisar logs
  - Colores hex en vez de design system: gcids no sincronizados → Paso 1.5
  - Módulos no aparecen: tipo divi/* incorrecto → verificar blocks-dictionary.md
  - Preset ignorado silenciosamente: nombre no existe → verificar _design_presets.json
  - Build falla silencioso: usar --out para ver schema raw antes de deploy

SECCIÓN 5 — EL CATÁLOGO: CÓMO USARLO BIEN Y CÓMO NO USARLO
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Los 877 templates del catálogo (workspace/sections/catalog/) son layouts de
Divi Plus probados en producción — una mina de oro de composiciones reales.
El problema nunca fue el catálogo: fue cómo orchestrate_page.php los inyecta.

USO CORRECTO (por el agente, manualmente):

  1. Buscar layouts relevantes:
       python DAW_bundle/workspace/automation/search_catalog.py --query "timeline" --limit 3
     Devuelve paths con score semántico. Abrir el JSON de la coincidencia.

  2. Inspeccionar el JSON:
       - Estructura de filas y columnas (column_structure, anidación)
       - Proporciones de spacing (márgenes, paddings, alturas)
       - Patrones de composición de módulos (qué tipos se combinan cómo)
       - Ritmo visual (qué elemento domina cada fila)

  3. Extraer lo útil y TRADUCIR OBLIGATORIAMENTE:
       Hex hardcodeados         → {{design:color:ink}}, {{design:color:accent}}, etc.
       Imágenes de demo         → assets reales del cliente o {{SITE_URL}}/wp-content/uploads/...
       et_pb_* shortcodes       → módulos divi/* equivalentes
       Clases CSS personalizadas → decoration nativa (spacing, border, boxShadow)
       Valores de spacing en %  → {{design:space:lg}}, {{design:space:2xl}}, etc.

  4. El resultado es JSON propio con tokens, no una copia del catálogo.

USO INCORRECTO (genera resultados rotos):

  ✗ Dejar que orchestrate_page.php elija e inyecte del catálogo automáticamente
    → trae colores RGB mapeados incorrectamente, imágenes de guardería en sitio
      de biblioteca, slots vacíos (button_text: ""), parser YAML frágil

  ✗ Copiar el JSON del catálogo directo al page-def sin traducir
    → hex hardcodeados, imágenes de demo ajenas, et_pb_* que no compila

  ✗ compose_page.php directamente
    → fill_slots() rellena con "" los slots no resueltos

  ✗ acerca-de.json como plantilla
    → paradigma legacy ROTO: divi/placeholder, slots vacíos, catálogo sin traducir

  ✗ mysql_remote.bat → deshabilitado
